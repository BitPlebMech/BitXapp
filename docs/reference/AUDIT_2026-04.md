### V2 — `settings.js` reads `localStorage` directly for storage info (Rule 1)
**Status: STILL PRESENT — not fixed**

**File:** `js/modules/settings/settings.js` · line 68

```js
const raw = localStorage.getItem('super_app_v1') || '';
const kb  = (new Blob([raw]).size / 1024).toFixed(1);
const pct = ((raw.length / 5120000) * 100).toFixed(1);
```

**Severity: Cosmetic** — read-only, no data corruption risk. But it violates the rule and it duplicates logic that already exists in `App.State.storageInfo()`. There is an additional inconsistency: `storageInfo()` in `state.js` uses `raw.length * 2 / 1024` (UTF-16 byte estimate), while `_updateStorageInfo()` in `settings.js` uses `new Blob([raw]).size / 1024` (UTF-8 byte count). These produce different numbers for the same data. The `Blob` approach is more accurate, so the fix is to move that logic into `storageInfo()`.

**Minimal fix:**

Update `storageInfo()` in `state.js` to return a structured object (or update the signature to return the same string `settings.js` builds):
```js
// state.js — replace existing storageInfo()
function storageInfo() {
  const raw = localStorage.getItem(STORAGE_KEY) || '';
  const kb  = (new Blob([raw]).size / 1024).toFixed(1);
  const pct = ((raw.length / 5120000) * 100).toFixed(1);
  return { kb, pct, display: `${kb} KB (~${pct}% of 5 MB)` };
}
```

Then in `settings.js` `_updateStorageInfo()`:
```js
function _updateStorageInfo() {
  const info = window.App.State.storageInfo();
  if (el('stg-storage'))
    el('stg-storage').textContent = `localStorage: ${info.display} used`;
}
```

Note: `portfolio.js` `updateStorageStatus()` also calls `window.App.State.storageInfo()` at line 126, but only uses the returned string directly (`const info = window.App.State.storageInfo()`). Update that call site too after the shape change.

### V3 — `settings.js` calls `App.Portfolio.toggleTheme()` directly (Rules 3 + 4)
**Status: STILL PRESENT — not fixed**

**File:** `js/modules/settings/settings.js` · lines 337–345

```js
function toggleTheme() {
  if (typeof window.App.Portfolio?.toggleTheme === 'function') {
    window.App.Portfolio.toggleTheme();   // ← cross-module call
  } else {
    // fallback writes directly to portfolio.settings.theme ...
    const s = window.App.State.getPortfolioData();
    if (s.settings) { s.settings.theme = next; window.App.State.setPortfolioData(s); }
  }
}
```

**Severity: Structural** — `settings.js` has a direct runtime dependency on `App.Portfolio`. If Portfolio is not yet initialised (or is removed), the fallback branch writes theme into `portfolio.settings`, which is namespace leakage (V9).

**Minimal fix** (two parts, must be done together):

**Part A — move `toggleTheme()` and `applyTheme()` to `app-shell.js`:**
```js
// In app-shell.js — add a real toggleTheme()
function toggleTheme() {
  const s    = window.App.State.getPortfolioData();
  const next = (s.settings?.theme || 'dark') === 'dark' ? 'light' : 'dark';
  s.settings.theme = next;
  window.App.State.setPortfolioData(s);
  applyTheme();   // already exists in app-shell.js
  // sync theme icon in portfolio header if it exists
  const sun  = document.getElementById('theme-icon-sun');
  const moon = document.getElementById('theme-icon-moon');
  if (sun)  sun.style.display  = next === 'dark'  ? '' : 'none';
  if (moon) moon.style.display = next === 'light' ? '' : 'none';
}
// Add to return block: toggleTheme
```

**Part B — update `settings.js`:**
```js
function toggleTheme() {
  window.App.Shell.toggleTheme();   // ← one line, no cross-module call
}
```

**Part C — update `portfolio.js`:** Keep `portfolio.js`'s own `toggleTheme()` and `applyTheme()` for internal use (they update the icon state correctly), but make `toggleTheme()` delegate to Shell to avoid duplicate state-write logic:
```js
function toggleTheme() {
  window.App.Shell.toggleTheme();   // Shell owns the write
  applyTheme();                     // local icon sync (reads from state after Shell wrote)
}
```

Note: This does NOT yet move theme storage out of `portfolio.settings` (that is V9). Do V3 and V9 together when the full refactor is scheduled.

### V7 — `settings.js` calls `App.Portfolio` internals directly (Rule 3)

### V14 — `App.Shell.triggerGistLoad()` re-renders by calling `window.App.EmberUI.render()` directly (Rule 4)

**File:** `js/core/app-shell.js` · lines 420–421

```js
if (window.App.EmberUI?.render)   window.App.EmberUI.render();
if (window.App.HabitsUI?.render)  window.App.HabitsUI.render();
```

**Severity: Structural** — Shell is calling module-UI functions directly. The correct pattern is for Shell to call the module's public `render()` or to re-run the module's `init()` flow. `App.EmberUI` is an internal sub-component of the Ember module; Shell should call `App.Ember.render()` (if Ember exposes one) or re-trigger `switchModule('ember')` to let the module handle its own re-render.

`App.Habits` does expose a `render` delegate via `_render()` → `App.HabitsUI.render()` internally, but that isn't the public surface. Looking at habits.js, there is no `render` in the public exports.

**Minimal fix:** Add a public `render()` function to `App.Ember` and `App.Habits` that each delegate to their respective UI:
```js
// In habits.js exports:
render: _render,

// In ember.js exports:
render: () => window.App.EmberUI?.render?.(),
```

Then update Shell:
```js
if (window.App.Ember?.render)  window.App.Ember.render();
if (window.App.Habits?.render) window.App.Habits.render();
```

| Mistake | Consequence | Rule |

Add to **Section 6 — Registering a new module (checklist)**:

- [ ] Wire the module's Gist Save button to `App.Shell.triggerGistSave()`, **not** `module.triggerGistSave()` — unless you intentionally want a module-only partial save
- [ ] Expose a public `render()` function in the module's return block so Shell can trigger a re-render after Gist load without reaching into the module's UI layer directly
- [ ] If the module has any "last run" or "last sent" timestamps, store them inside the module's namespace in `state.js`, never as raw `localStorage` keys

## B. New Module Checklist Additions

The following items are missing from the current checklist in `MODULE_RULES.md §6`:

1. **Expose a public `render()` method.** Shell needs to call it after Gist load. Without it, Shell must reach into the module's UI sub-layer, which is an isolation violation.

2. **Wire the "Save" button to `App.Shell.triggerGistSave()`.** Document a clear decision: each module's header Save button should call `App.Shell.triggerGistSave()` (saves all) rather than `module.triggerGistSave()` (saves one). If per-module save is needed, add a secondary button, but always keep a "Save All" path.

3. **Never store module state outside `super_app_v1`.** Any per-module bookkeeping (last-sent date, last-viewed item, cached value) must live in the module's namespace, accessed via `App.State`.

4. **Test the `App.Shell.triggerGistLoad()` path explicitly.** After adding a module, manually trigger a full Gist load and verify the module's data is fully restored. The "empty books tab" bug was only caught in production.

## C. The Rules Card

Paste this comment block at the top of every new module file, immediately after `'use strict';`:

```js
/**
 * MODULE RULES — read before editing (full spec: docs/MODULE_RULES.md)
 *
 * 1. STATE     — Read/write only your own namespace via App.State.getXxxData()
 *                / setXxxData(). NEVER call localStorage directly.
 *
 * 2. GIST      — Use only your module's Gist function (App.Gist.saveXxxData /
 *                loadXxxData). One Gist file per module. triggerGistLoad() must
 *                restore ALL fields of your namespace, not just some.

 * 4. SHELL     — Wire Save buttons to App.Shell.triggerGistSave() for "save all".
 *                Expose a public render() so Shell can re-render you after load.
 *                Toast → App.Shell.toast(). Confirm → App.Shell.confirmAction().
 *
 * 5. LOAD ORDER — script tag goes before settings.js in index.html.
 *                 data file → business logic file → UI file.
 */
```
